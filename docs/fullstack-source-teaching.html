一Maven 多模块架构

1.1 为什么要拆模块？

核心原则：编译期隔离，运行期聚合。每个模块只暴露最小 API 面，内部实现细节对外不可见。

项目共 5 个 Maven 模块，由父 POM 统一管理版本：

blog-api启动模块 · 唯一 @SpringBootApplication
blog-auth认证授权 · Security + JWT
blog-core核心业务 · 文章/分类/标签/评论
blog-file文件服务 · 上传/管理
blog-common公共基础 · R/异常/Redis/工具

1.2 为什么不是微服务？

维度单体（本项目）微服务
部署复杂度一个 JAR，一条命令启动需要 K8s/注册中心/配置中心/网关
开发效率IDE 一键启动，断点调试方便多服务联调，本地环境复杂
事务管理本地事务 @Transactional 简单可靠分布式事务（Seata/SAGA），复杂度高
运维成本一台服务器即可至少 3-5 台 + 中间件集群
适用场景个人博客、企业官网、中小型系统大型电商、多团队协作、高并发系统
本项目的选择✓ 单体过度设计，ROI 极低

架构原则：用 Maven 模块实现代码级解耦（编译期），运行时仍然是单体（一个 JVM 进程）。这兼顾了代码组织和运维简单性。如果未来需要拆分，模块边界已经画好，只需把 blog-auth/blog-core/blog-file 各自独立部署并改 HTTP 调用即可。

1.3 父 POM 的版本统一策略

所有第三方依赖版本号集中在父 POM 的 <dependencyManagement> 中定义。子模块声明依赖时无需指定版本，自动从父 POM 继承。这样做的好处：

版本一致性

不会出现 blog-auth 用 jjwt 0.11、blog-core 用 jjwt 0.12 的冲突

升级方便

升 SpringBoot 版本只需改父 POM 一处，所有子模块自动生效

传递依赖安全

Maven 仲裁机制在父 POM 明确定义下，避免"幽灵版本"问题

<!-- 父 POM 版本集中营 -->
<properties>
  <java.version>17</java.version>
  <mybatis-plus.version>3.5.5</mybatis-plus.version>
  <jjwt.version>0.12.5</jjwt.version>
  <mapstruct.version>1.5.5.Final</mapstruct.version>
</properties>

三认证授权系统

3.1 Spring Security 配置思想

三关两不：关闭 Session（无状态）、关闭 CSRF（非 Cookie 认证）、API 分两类（/api/public/** 放行、/api/admin/** 需 ADMIN 角色）。

@Bean
public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
    http
        .csrf(AbstractHttpConfigurer::disable)           // 关 CSRF
        .sessionManagement(s -> s
            .sessionCreationPolicy(SessionCreationPolicy.STATELESS)) // 关 Session
        .authorizeHttpRequests(auth -> auth
            .requestMatchers("/api/admin/**").hasRole("ADMIN")  // 后台: 需 ADMIN
            .requestMatchers("/api/public/**").permitAll()      // 公开: 无需登录
            .anyRequest().authenticated()
        )
        .addFilterBefore(jwtAuthFilter, UsernamePasswordAuthenticationFilter.class);
}

3.3 JWT 双 Token 机制

特性AccessTokenRefreshToken
有效期15 分钟7 天

3.4 Token 轮换 (Token Rotation) 机制

RefreshToken 轮换流程（防重放攻击）
1. 用户携带旧 RefreshToken 请求 /refresh
↓
2. 校验签名 + 过期 + type="refresh"
↓
3. SHA-256(旧Token) vs Redis 中存储的 hash — 不匹配？
↓ 匹配
4. 删除旧 RefreshToken（Redis）
↓
5. 生成新 AccessToken（15min）+ 新 RefreshToken（7d）
↓
6. 新 RefreshToken 的 hash 写入 Redis

安全价值：如果有人窃取了旧的 RefreshToken，当合法用户用最新 RefreshToken 刷新后，旧 Token 的 hash 与 Redis 中不匹配，系统会检测到异常并主动删除所有 RefreshToken，强制重新登录。这就是 RefreshToken 轮换（Token Rotation）防重放的核心。

3.5 登出流程：黑名单机制

登出时做了两件事：① 删除 Redis 中的 RefreshToken（Key = blog:token:refresh:{userId}）；② 将 AccessToken 的 SHA-256 哈希加入黑名单（Key = blog:token:blacklist:{sha256}），TTL 等于该 Token 的剩余有效时间（最长 15 分钟）。

为什么需要黑名单？因为 JWT 是无状态的——后端发出后无法"撤回"。即使登出了，只要 Token 没过期，它本身仍然是有效的。黑名单就是用来补这个漏洞的。

四blog-common 深度解析

4.1 R<T> — 统一响应体的设计哲学

为什么不用 HTTP 状态码判断成败？HTTP 状态码只能区分 2xx/4xx/5xx，无法表达业务语义。同一个 400 状态码可能对应"参数错误"、"用户名已存在"、"余额不足"等多种情况。用 R<T>.code 的 ResultCode 枚举，前端不需要解析 HTTP 状态码，统一读 response.data.code。

public class R<T> implements Serializable {
    private Integer code;      // 业务状态码
    private String  message;   // 提示信息
    private T       data;      // 响应数据（泛型）
    private Long    timestamp; // 响应时间戳
}

4.2 ResultCode 枚举 — 为什么不用字符串？

状态码用枚举而非字符串的三个原因：① 类型安全——编译器检查，不会打成 "UNAUTHORISED"；② IDE 自动补全——不需要记住状态码字符串；③ 统一语义——200=成功, 401=未登录, 403=无权限, 404=不存在，前端可以根据 code 做统一拦截（如 code=401 时弹出登录框）。

4.3 GlobalExceptionHandler — 为什么 Controller 不写 try-catch

Controller 抛出异常
↓
@RestControllerAdvice 拦截
↓
转换为 R.fail(code, message) 返回

设计原则：Controller 不写 try-catch，异常统一抛出。由 @RestControllerAdvice 的 GlobalExceptionHandler 集中处理，保证：① Controller 代码简洁；② 错误格式全部统一为 R 格式；③ 异常日志集中管理（log.warn/log.error）。

4.4 TraceIdFilter — 链路追踪实现

每个请求进入时，生成唯一 traceId 并存入 SLF4J MDC（Mapped Diagnostic Context）。MDC 是线程绑定的——同一个线程内的所有日志自动带上 traceId。请求结束后在 finally 块中清除 MDC，防止线程池复用导致数据串扰。

优先读取前端传的 X-Trace-Id Header，没有则生成新的 16 位 UUID。同时把 traceId 通过 Response Header 回传给前端，前端可以在控制台看到每个请求的 traceId，方便排查。

4.5 RedisKeys — 为什么要集中管理 Key

所有 Redis Key 定义在 RedisKeys 工具类中，格式为 blog:{功能}:{参数}：

六分类模块源码教学

6.1 树形结构 — parentId 设计

分类表通过 parentId 字段实现树形结构：parentId = NULL 的为根分类，parentId = 某分类ID 的为子分类。这是经典的邻接表模型——最简单、最直观的树形数据存储方式。

6.2 为什么在内存中构建树？

分类数据量小（通常 < 100 条），全部加载到内存后用 Java Stream 构建树，比数据库递归查询快几个数量级。实现思路：

// 1. 查出所有分类
List<Category> categories = categoryMapper.selectList(wrapper);

// 2. 按 parentId 分组
Map<Long, List<CategoryVO>> childrenMap = vos.stream()
    .filter(vo -> vo.getParentId() != null)
    .collect(Collectors.groupingBy(CategoryVO::getParentId));

// 3. 取出根节点，递归挂载子节点
List<CategoryVO> tree = vos.stream()
    .filter(vo -> vo.getParentId() == null)
    .peek(vo -> vo.setChildren(childrenMap.getOrDefault(vo.getId(), List.of())))
    .toList();

6.3 删除保护

删除分类前做了两道保护：① 检查是否有子分类，有则拒绝删除；② 检查是否有关联文章，有则拒绝删除。这是数据完整性的最后防线——数据库外键可能被禁用或未配置，代码层面的校验更可靠。

7.2 延迟双删策略

为什么要删两次？写操作时，"先删缓存 → 更新DB → 再删缓存"是为了防止并发场景下的数据不一致：如果 A 线程更新了 DB 但还没删缓存，B 线程在这期间读到了旧缓存数据。延迟 500ms 再删一次，给并发读留出"写完 DB"的时间窗口，最终一致性概率极大提高。

7.3 缓存穿透/击穿/雪崩防护

问题本项目的应对
缓存穿透
查询不存在的数据		对于不存在的数据，Service 层直接抛 NotFoundException，不缓存 null
缓存击穿

八项目工程化思想

为什么统一异常？

Controller 不写 try-catch。异常由 @RestControllerAdvice 统一拦截。错误格式统一，日志集中管理，Controller 代码薄如纸。

为什么统一返回 R<T>？

前后端分离场景下，HTTP 状态码不够用。R 的 code 字段可以承载任意业务语义，前端不需要根据 HTTP 状态码做分支判断。

为什么 Controller 不写业务逻辑？

Controller 的职责是"接参数 → 调 Service → 返回结果"。业务逻辑在 Service 中，事务边界也在 Service。如果 Controller 写业务，测试和复用都困难。

为什么 DTO/VO 分离？

DTO（入参）和 VO（出参）面向不同方向，字段不同。入参不需要 id/createTime，出参可能需要脱敏或组合字段。分离后各自变化互不影响。

为什么事务在 Service？

@Transactional 标注在 Service 方法上，保证一个业务操作内的多个数据库操作要么全成功要么全回滚。Controller 不应该关心事务边界。

为什么统一配置？

版本号集中在父 POM，Redis Key 集中在 RedisKeys，状态码集中在 ResultCode。需要改的时候只改一处，不会遗漏。

9.3 API 封装层 — 双模 fetch 设计

核心问题：Astro 页面在服务端（Node.js SSR）和客户端（浏览器）都会执行 fetch()。Node.js 的 fetch 无法解析相对路径 /api/public/articles，必须用绝对 URL http://localhost:8080/api/public/articles。而浏览器端用相对路径即可（Vite proxy 自动转发）。

async function request<T>(
  method: string, path: string, options?: RequestOptions
): Promise<R<T>> {
  const url = new URL(BASE + path, 'http://localhost');
  // ★ 环境检测
  const isSSR = typeof window === 'undefined';
  const fullPath = isSSR
    ? `http://localhost:8080${url.pathname}${url.search}`  // SSR: 绝对URL
    : `${url.pathname}${url.search}`;                         // 浏览器: 相对路径+Vite proxy
  const res = await fetch(fullPath, { method, headers, body });
  // ★ 双重校验
  if (!res.ok) throw new Error(json.message || '请求失败');
  if (json.code !== 200) throw new Error(json.message || '业务错误');
}

Token 刷新机制：accessToken 过期（15min）后，前端调用 refreshApi(refreshToken)。后端验证 refreshToken 签名 + Redis 中匹配 → 返回新的 accessToken + refreshToken → 前端更新 Cookie。用户无感。

路由权限：管理端页面（/admin/*）中的 Vue 组件通过 client:load 在浏览器侧渲染，API 调用自动附带 Cookie 中的 Token。后端 /api/admin/** 要求 hasRole("ADMIN")，未授权请求返回 403。

9.6 前端工程化思想

为什么不能页面直接请求接口？

直接把 fetch 写在页面里会导致：Token 注入逻辑散落各处、错误处理不一致、API 路径硬编码。封装成 api.ts 后，31 个 API 函数共享同一套请求逻辑。

为什么 Layout 分层？

BaseLayout → PublicLayout/AdminLayout 三层继承。Base 负责 HTML shell 和 SEO meta，Public 加 Header/Footer，Admin 加侧边栏。需要改全局样式只改 Base，需要改管理端导航只改 Admin。

为什么组件化？

ArticleCard、Pagination、TagPill 等组件在多个页面复用。改一个文件，所有用到的地方自动更新。组件 Props 定义了明确的输入接口。

为什么 SSR + 客户端混合？

公共页面（首页/详情/搜索）在服务端渲染 HTML → SEO 友好 + 首屏快。管理端 Vue 组件通过 client:load 在浏览器侧水合 → 交互灵活。这是 Astro "岛屿架构"的精髓。
