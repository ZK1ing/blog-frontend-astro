1.3关键设计决策 — 为什么这样设计

为什么拆 5 个模块而不是一个大 jar？

编译期隔离 = 代码边界强制。blog-auth 不能访问 blog-core 的 ArticleService，blog-core 不能访问 blog-auth 的 JwtTokenProvider（除非显式声明依赖）。模块边界就是代码规范——不会出现"写着写着就相互引用成一团乱麻"的情况。

为什么是单体而不是微服务？

个人博客的流量不需要微服务的弹性伸缩能力，但微服务的复杂度（网络延迟、分布式事务、服务发现、配置中心）是实打实的。用 Maven 模块实现"代码级解耦"，保留"单体部署的运维简单性"，是最佳性价比选择。

为什么父 POM 统一版本？

5 个模块如果各自声明依赖版本，升级时可能出现 blog-auth 用 jjwt 0.11、blog-core 用 jjwt 0.12 的冲突。父 POM 的 <dependencyManagement> 集中管理，子模块只声明 GAV 不写版本号，一改全改。

为什么把 auth 独立出来？

认证授权是横切关注点，和具体业务（文章/分类）无关。独立后：① auth 可以单独升级安全策略而不影响业务；② 未来如果需要把认证改为 OAuth2/OIDC，只改 blog-auth 一个模块；③ 如果要做多系统 SSO，blog-auth 可以直接抽成独立服务。

2.1 com.blog.common.result — 统一响应包

R<T>.java — 为什么所有接口返回同一个格式？

前后端分离架构中，后端每个接口的响应格式必须统一，前端才能写统一的拦截器和错误处理。如果有的接口返回 {"data":...}、有的返回 {"error":"..."}、有的直接抛 500 页面，前端就要针对每种情况写不同的处理逻辑——这是灾难。

R<T> 解决的就是这个问题：

  • code（Integer）：业务状态码。200=成功，401=未登录，403=无权限，404=不存在。前端根据 code 做统一拦截（如 code=401 弹登录框）
  • message（String）：人类可读的提示信息，可以直接展示给用户
  • data（T 泛型）：实际数据。可以是 ArticleDetailVO、Page<ArticleListVO>、或者 null
  • timestamp（Long）：响应时间戳，便于排查问题

静态工厂方法的设计思想：

// 成功 — 只需关心数据，code 和 message 自动填充
R.ok(articleDetailVO);         // code=200, message="操作成功"

// 失败 — 传入 ResultCode 枚举，类型安全
R.fail(ResultCode.UNAUTHORIZED);  // code=401, message="未登录或 Token 已过期"
R.fail(ResultCode.NOT_FOUND);     // code=404

// 也可以自定义消息
R.fail(400, "用户名已存在");      // 自由定制的错误信息

为什么不直接用 HTTP 状态码判断成败？ HTTP 状态码只有大约 20 个常用值，无法表达丰富的业务语义。而且 HTTP 状态码是传输层的概念，业务状态码是应用层的概念——两者应该分离。R<T>.code 可以承载任意业务语义，不限于 HTTP 的 4xx/5xx。

ResultCode.java — 状态码为什么用枚举？

状态码不用字符串 ("UNAUTHORIZED") 或魔法数字 (401)，而用枚举：

  • 编译器检查：打成 "UNAUTHORISED" 编译不通过
  • IDE 自动补全：不需要记忆状态码
  • 统一管理：所有状态码集中在一个文件，一眼看清项目有哪些错误类型
  • 每个枚举项自带 message：code 和 message 绑定，不会出现 code=401 但 message 写着 "参数错误" 的混乱
public enum ResultCode {
    SUCCESS(200, "操作成功"),          // 成功
    BAD_REQUEST(400, "参数错误"),     // 参数校验失败
    UNAUTHORIZED(401, "未登录或 Token 已过期"), // 认证失败
    FORBIDDEN(403, "无权限访问"),     // 已登录但权限不足
    NOT_FOUND(404, "资源不存在"),     // 数据查不到
    TOO_MANY_REQUESTS(429, "请求过于频繁"), // 触发限流
    INTERNAL_ERROR(500, "服务器内部错误"),  // 未知异常兜底
    // ... 更多状态码
}

2.2 com.blog.common.exception — 异常体系

为什么要自定义异常？

Java 标准异常（IllegalArgumentException、RuntimeException）无法携带业务状态码。如果 Controller 层抛了 RuntimeException，全局异常处理器只能统一返回 500——但实际可能是 404（资源不存在）、401（未登录）、403（无权限）。

自定义异常类携带 ResultCode，全局异常处理器根据不同的异常类型返回不同的 HTTP 状态码 + 业务状态码。

// ★ 异常继承树（4 层）
RuntimeException
  └── BusinessException        // 基类，携带 ResultCode
        ├── UnauthorizedException  // 401 — 未登录/Token无效
        ├── ForbiddenException     // 403 — 已登录但无权限
        ├── NotFoundException     // 404 — 资源不存在
        └── BusinessException     // 400 — 通用业务异常（可自定义消息）

BusinessException 的两个构造器体现了灵活性：

  • new BusinessException(ResultCode.UNAUTHORIZED) → 使用状态码的默认消息
  • new BusinessException(ResultCode.UNAUTHORIZED, "用户名或密码错误") → 保留状态码，覆盖消息文本
  • new BusinessException("分类下存在子分类，无法删除") → 默认 400，自定义业务语义

GlobalExceptionHandler.java — 为什么 Controller 不写 try-catch？

核心思想：Controller 代码里不应该出现 try-catch。异常是"非正常流程"，应该由框架统一处理。Controller 只需要写"正常流程"的代码——接参数、调 Service、返回 R.ok()。任何异常（参数校验失败、认证失败、业务异常、未知异常）都由 @RestControllerAdvice 统一拦截并转换为 R 格式。

2.3 com.blog.common.filter — TraceIdFilter 链路追踪

这个类是干什么的？

每个 HTTP 请求进入后端时，TraceIdFilter 给它分配一个全局唯一的 traceId（16 位 UUID），并通过 SLF4J 的 MDC（Mapped Diagnostic Context） 机制注入到当前线程的所有日志输出中。

实际效果：一次请求可能经过 Filter → Controller → Service → Mapper 多个层，每层都可能打日志。没有 traceId，你无法把分散在几百行日志里的同一个请求关联起来。有了 traceId，grep 一下就能看到完整调用链。

// 日志输出效果对比

// 没有 traceId（无法关联）
2026-05-14 10:23:45 INFO  - 接收到文章查询请求
2026-05-14 10:23:45 INFO  - 查询文章列表，pageNum=1
2026-05-14 10:23:45 INFO  - 返回 10 条文章

// 有 traceId（grep a1b2c3d4 看到完整链路）
2026-05-14 10:23:45 [a1b2c3d4] INFO  - 接收到文章查询请求
2026-05-14 10:23:45 [a1b2c3d4] INFO  - 查询文章列表，pageNum=1
2026-05-14 10:23:45 [a1b2c3d4] INFO  - 返回 10 条文章

关键细节：

  • @Order(Ordered.HIGHEST_PRECEDENCE + 10) — 确保在所有 Filter 中第一个执行，traceId 从请求进入的第一刻就存在
  • 优先读取前端传的 X-Trace-Id Header — 如果前端已经生成了 traceId，后端复用，实现全链路追踪（浏览器 → 后端 → 数据库 同一 traceId）
  • 响应 Header 中也返回 X-Trace-Id — 前端可以拿到这个值，出问题时告诉后端 "traceId 是 xxx"
  • finally { MDC.clear(); } — 绝对不能忘！Tomcat 线程池复用线程，不清 MDC 会导致下一个请求的日志带上上一个请求的 traceId

2.4 com.blog.common.config — 配置类

MyBatisPlusConfig.java — 两个关键 Bean

这个类做了两件事：

  • 分页插件（MybatisPlusInterceptor + PaginationInnerInterceptor）：MyBatis-Plus 的分页不是自动生效的，必须注册这个插件。它拦截 SQL 执行，自动在查询前加 COUNT 语句、查询后加 LIMIT 语句。如果不注册，Page 对象不会自动分页。
  • 自动填充处理器（MetaObjectHandler）：所有继承 BaseEntity 的实体，在 insert 时自动填充 createTime 和 updateTime，在 update 时自动填充 updateTime。这样就不需要在每个 Service 方法里手动 set 时间——一处配置，全局生效。

RedisConfig.java — 序列化配置

Spring Boot 默认的 RedisTemplate 使用 JDK 序列化，存入 Redis 的 Key 和 Value 都是二进制乱码，无法在 Redis CLI 中直接查看。这个配置类改为：

  • Key → StringRedisSerializer（人类可读的字符串）
  • Value → GenericJackson2JsonRedisSerializer（JSON 格式，含类型信息可反序列化）
  • ObjectMapper 注册 JavaTimeModule（支持 LocalDateTime 序列化）

JacksonConfig.java — 日期格式统一

全局设置 Jackson 的日期序列化规则：所有 LocalDateTime 字段输出为 "yyyy-MM-dd HH:mm:ss" 格式，null 值不输出（NON_NULL）。一处配置，所有 Controller 返回的 JSON 都自动格式化日期。

WebMvcConfig.java — CORS + 拦截器注册

三个职责：① CORS 全局允许跨域（开发阶段方便）；② 注册 RateLimitInterceptor 拦截 /api/** 路径；③ 配置 /uploads/** 静态资源映射（上传的文件可以直接通过 URL 访问）。

Key 命名有层级结构：blog:{功能类别}:{子类别}:{参数}，冒号分隔形成逻辑层级，方便在 Redis 管理工具中按层级浏览。

ServletUtils.java — IP 获取工具

获取客户端真实 IP 的逻辑比看上去复杂——如果前面有 Nginx/代理，request.getRemoteAddr() 只能拿到代理的 IP。正确做法是先检查 X-Forwarded-For → X-Real-IP → getRemoteAddr()。

三blog-auth — 认证授权模块（12 个文件）

模块定位

📦 blog-auth 负责整个系统的身份认证和访问控制。它依赖 blog-common（需要 R 格式返回、需要 Redis），但不依赖 blog-core。
核心能力：① 用户登录/登出/刷新 Token ② Spring Security + JWT 无状态认证 ③ Token 黑名单 + RefreshToken 轮换防重放。

3.1 SecurityConfig — Spring Security 配置（整个安全体系的总控）

这是整个安全体系的"指挥部"。看看它做了什么决定：

@Bean
public SecurityFilterChain filterChain(HttpSecurity http) {
    http
        // ① 关闭 CSRF：因为是前后端分离 + JWT 认证，不存在 CSRF 攻击面
        .csrf(AbstractHttpConfigurer::disable)

        // ② 无状态会话：不创建 HttpSession，每次请求都带 JWT
        .sessionManagement(s -> s
            .sessionCreationPolicy(SessionCreationPolicy.STATELESS))

        // ③ 权限规则：白名单 + 角色要求
        .authorizeHttpRequests(auth -> auth
            .requestMatchers("/api/admin/**").hasRole("ADMIN")  // 管理端→需ADMIN角色
            .requestMatchers("/api/public/**").permitAll()      // 公开接口→任何人
            .anyRequest().authenticated()                          // 其余→至少登录
        )

        // ④ 插入 JWT 过滤器（在 Spring Security 默认认证过滤器之前）
        .addFilterBefore(jwtAuthFilter,
            UsernamePasswordAuthenticationFilter.class)

        // ⑤ 异常处理：认证失败→401 JSON，授权失败→403 JSON
        .exceptionHandling(ex -> ex
            .authenticationEntryPoint(/* 返回 R.fail(401) 的 JSON */)
            .accessDeniedHandler(/* 返回 R.fail(403) 的 JSON */)
        );
}

设计决策一：为什么关闭 CSRF？
CSRF 攻击的前提是浏览器自动携带 Cookie 发送请求。但本项目用 JWT 放在 Authorization Header 中，浏览器不会自动发送 Header——攻击者即使构造了恶意请求，也无法带上 JWT。所以 CSRF 保护是多余的。

设计决策二：为什么是 STATELESS？
每个请求独立携带 JWT，服务器不需要记住"这个用户刚才登录过"。服务器可以随时重启而不丢失任何会话状态。多台服务器部署时也不需要 Session 同步。

设计决策三：为什么异常处理要返回 JSON 而不是重定向到登录页？
前后端分离架构下，前端不是从后端返回的 HTML 页面，而是独立的 JS 应用。后端返回 302 重定向到 /login，前端 Ajax 请求跟过去也拿不到页面。正确的做法是返回 JSON 格式的错误信息，前端根据 code=401 来决定跳转登录页。

3.2 JwtTokenProvider — JWT 工具（Token 的工厂）

这个类是 JWT 操作的唯一入口。所有 Token 的创建、解析、校验都经过这里。

// ★ 常量定义
private static final long ACCESS_EXPIRATION  = 15 * 60 * 1000L;    // AccessToken 15分钟
public static final long REFRESH_EXPIRATION = 7 * 24 * 60 * 60 * 1000L; // RefreshToken 7天

// ★ 密钥初始化 — 从配置文件读取，确保至少 256 位
public JwtTokenProvider(@Value("${blog.jwt.secret:...}") String secret) {
    byte[] keyBytes = new byte[32];  // 256位 = 32字节
    System.arraycopy(secret.getBytes(), 0, keyBytes, 0,
        Math.min(secret.getBytes().length, 32));
    this.secretKey = Keys.hmacShaKeyFor(keyBytes);
}

AccessToken 的载荷（Claims）设计：

  • userId：用户 ID，后续通过 SecurityContext 获取当前用户
  • role：角色（ADMIN），用于 @PreAuthorize 注解和 Security 的 hasRole 判断
  • type: "access"：区分 Token 类型的关键字段。防止有人拿 RefreshToken 当 AccessToken 用
  • iss: "blog-server"：签发者标识，防止跨应用混用 Token
  • exp：过期时间（15 分钟后）

RefreshToken 和 AccessToken 的区别：

  • RefreshToken 不包含 role（它不用于授权，只用于刷新）
  • RefreshToken 的 type 是 "refresh"（JwtAuthFilter 检测到 type=refresh 会拒绝）
  • RefreshToken 的有效期是 7 天而不是 15 分钟
  • RefreshToken 的哈希存储到 Redis，AccessToken 不存（自包含验证）

为什么 AccessToken 不存 Redis？因为它的有效期只有 15 分钟，且需要高频访问（每个请求都要验证）。如果每次请求都查 Redis，Redis 就成了性能瓶颈。JWT 的设计初衷就是"自包含认证"——Token 本身携带足够信息，无需查数据库或缓存即可验证。黑名单是例外——只查登出用户的 Token。

3.3 JwtAuthFilter — JWT 认证过滤器（请求的守门人）

每个 HTTP 请求到达时，JwtAuthFilter 负责：从 Header 取 Token → 验证签名和有效期 → 检查黑名单 → 设置 SecurityContext。6 步流程：

JwtAuthFilter.doFilterInternal 执行流程
HTTP 请求进入
↓
步骤 1：extractToken() — 从 Authorization Header 提取 Bearer Token
↓ 无 Token？→ 放行，让后续 Security 权限校验拦截
步骤 2：validateToken() — 验证签名（防篡改）+ 验证过期时间
↓
步骤 3：getTokenType() — 只放行 type="access"，拒绝 type="refresh"
↓
步骤 4：检查 Redis 黑名单 — hasKey(blog:token:blacklist:{sha256})
↓ 不在黑名单
步骤 5：从 JWT Claim 提取 userId + role → 构建 UsernamePasswordAuthenticationToken
↓
步骤 6：SecurityContextHolder.getContext().setAuthentication(authentication)
↓
chain.doFilter() → 请求继续传递到 Controller

为什么继承 OncePerRequestFilter？ 保证每个请求只执行一次过滤。普通的 Filter 在一次请求中可能被调用多次（比如 forward/include），而 JWT 校验只需要做一次。

为什么 JwtAuthFilter 不做数据库查询？ JWT 的自包含特性：userId 和 role 都编码在 Token 中，验证签名即可确认这些信息没有被篡改。如果每次请求都去查 user 表，JWT 的性能优势就荡然无存了。

3.4 AuthServiceImpl — 登录/登出/刷新 源码级讲解

登录流程

public LoginVO login(LoginDTO dto) {
    // 1. LambdaQueryWrapper 按 username 查用户
    User user = userMapper.selectOne(
        new LambdaQueryWrapper<User>()
            .eq(User::getUsername, dto.getUsername())
    );
    if (user == null) throw new BusinessException(UNAUTHORIZED, "用户名或密码错误");

    // 2. BCrypt 验密 — 注意：密码哈希是单向的，只能 matches 不能 decrypt
    if (!passwordEncoder.matches(dto.getPassword(), user.getPasswordHash()))
        throw new BusinessException(UNAUTHORIZED, "用户名或密码错误");

    // 3. 签发双 Token
    String accessToken  = jwtTokenProvider.createAccessToken(user.getId(), user.getRole());
    String refreshToken = jwtTokenProvider.createRefreshToken(user.getId());

    // 4. RefreshToken 存 Redis（存的是 SHA-256 哈希，不是明文）
    redisTemplate.opsForValue().set(
        RedisKeys.TOKEN_REFRESH + user.getId(),     // Key: blog:token:refresh:{userId}
        jwtTokenProvider.hashToken(refreshToken),    // Value: SHA-256 哈希
        Duration.ofMillis(REFRESH_EXPIRATION)        // TTL: 7天
    );

    return LoginVO.builder()
        .accessToken(accessToken).refreshToken(refreshToken)
        .userId(user.getId()).username(user.getUsername()).build();
}

为什么错误消息不区分"用户名不存在"和"密码错误"？这是安全最佳实践——如果告诉攻击者"用户名不存在"，攻击者就可以枚举系统中的用户名。统一返回"用户名或密码错误"可以防止用户枚举攻击。

登出流程

public void logout(String accessToken) {
    Long userId = SecurityUtils.getCurrentUserId();

    // 1. 删除 RefreshToken → 该用户下次刷新 Token 会失败
    redisTemplate.delete(RedisKeys.TOKEN_REFRESH + userId);

    // 2. AccessToken 哈希加入黑名单，TTL = Token 剩余有效时间
    //    因为 AccessToken 只有 15 分钟，黑名单也只需存最多 15 分钟
    Date expiration = jwtTokenProvider.getExpiration(accessToken);
    long remainingMs = expiration.getTime() - System.currentTimeMillis();
    if (remainingMs > 0) {
        redisTemplate.opsForValue().set(
            RedisKeys.TOKEN_BLACKLIST + jwtTokenProvider.hashToken(accessToken),
            "1", Duration.ofMillis(remainingMs));
    }
}

Token 刷新流程（含防盗检测）

public LoginVO refresh(RefreshDTO dto) {
    // 1. 校验旧 RefreshToken
    if (!jwtTokenProvider.validateToken(oldToken)) /* 签名无效或过期 */
    if (!"refresh".equals(jwtTokenProvider.getTokenType(oldToken))) /* 类型错误 */

    // 2. 从 Redis 取出存储的哈希
    String storedHash = redisTemplate.opsForValue().get(TOKEN_REFRESH + userId);

    // 3. SHA-256(用户传来的旧Token) vs Redis 中存储的哈希
    if (!jwtTokenProvider.hashToken(oldToken).equals(storedHash)) {
        // ★ 不匹配！意味着有人用了已经被轮换掉的旧 Token
        redisTemplate.delete(TOKEN_REFRESH + userId);  // 强制登出
        throw new UnauthorizedException("检测到令牌异常，请重新登录");
    }

    // 4. 轮换：删除旧 → 生成新双 Token → 新哈希存 Redis
    redisTemplate.delete(TOKEN_REFRESH + userId);
    String newAccess = jwtTokenProvider.createAccessToken(userId, role);
    String newRefresh = jwtTokenProvider.createRefreshToken(userId);
    redisTemplate.opsForValue().set(TOKEN_REFRESH + userId,
        jwtTokenProvider.hashToken(newRefresh), REFRESH_EXPIRATION);

    return LoginVO.builder().accessToken(newAccess).refreshToken(newRefresh)...;
}

public static Long getCurrentUserId() {

}

4.1 数据模型层：Entity → DTO → VO → Query

为什么有四层数据对象？

Entity（数据库映射）

Article.java、Category.java 等，带 @TableName、@TableId、@TableField 注解，直接对应数据库表结构。不能返回给前端——包含数据库实现细节，字段变更会影响 API。

DTO（前端→后端）

ArticleDTO、CommentDTO 等，带 @NotBlank、@Size 等校验注解。只包含前端需要传的字段。如 ArticleDTO 没有 viewCount（浏览量是服务端计算的）、没有 createTime（自动填充）。

VO（后端→前端）

ArticleListVO、ArticleDetailVO 等，面向展示优化。如 ArticleListVO 的 tags 是 List<String>（标签名），而 Entity 里根本没有 tags 字段（在关联表 article_tag 中）。VO 还承担数据组合的职责。

Query（查询条件）

ArticleQuery、CommentQuery 等，封装分页参数和筛选条件。和 DTO 分开是因为"查询条件"和"创建/更新数据"的字段完全不同。

4.2 ArticleServiceImpl — 最复杂的 Service（510 行）

Markdown 渲染 — Flexmark 静态单例

为什么把 Parser 和 HtmlRenderer 定义为 static final？因为这两个对象是线程安全且创建成本高的。如果每次请求都 new 一个 Parser，性能会严重下降。static final 保证全局只创建一次，所有请求共享。

private static final Parser MD_PARSER;
private static final HtmlRenderer MD_RENDERER;
static {
    MutableDataSet options = new MutableDataSet();
    MD_PARSER   = Parser.builder(options).build();
    MD_RENDERER = HtmlRenderer.builder(options).build();
}

分页查询 — 批量加载避免 N+1

toListVOs() 方法是性能优化的核心。一个典型的分页返回 10 篇文章，如果逐条查分类名和标签名，需要 1（文章列表）+ 10（分类）+ 10×N（标签）次数据库查询——即 N+1 问题。

该方法的处理方式是批量查询：

  1. 从 10 篇文章中提取所有 categoryId → 一次性 categoryMapper.selectBatchIds()
  2. 从 10 篇文章中提取所有 articleId → 一次性查 article_tag 表
  3. 从关联记录中提取所有 tagId → 一次性 tagMapper.selectBatchIds()
  4. 在内存中组装：article → categoryName + tagNames

总共只需 4 次数据库查询，而非 1 + 10 + 10×N 次。

浏览量原子递增 — 避免竞态条件

如果先 read 再 update，两个并发请求可能读到相同的 viewCount，都 +1 后写回——结果只增加了 1 而不是 2。

// ★ 错误写法（有竞态）
article.setViewCount(article.getViewCount() + 1);
articleMapper.updateById(article);

// ★ 正确写法（原子 SQL）
articleMapper.update(null, new LambdaUpdateWrapper<Article>()
    .eq(Article::getId, id)
    .setSql("view_count = view_count + 1"));  // 数据库层面原子递增

上一篇/下一篇导航

在文章详情页底部显示"上一篇"和"下一篇"链接。实现是查比当前文章 ID 小的第一篇 PUBLISHED 文章（prev）和比当前文章 ID 大的第一篇 PUBLISHED 文章（next）：

// 上一篇: ID 更小的第一篇
articleMapper.selectOne(new LambdaQueryWrapper<Article>()
    .eq(Article::getStatus, "PUBLISHED")
    .lt(Article::getId, articleId)   // ID 小于当前
    .orderByDesc(Article::getId)        // 按 ID 降序 → 取最接近的
    .last("LIMIT 1"));

4.3 CategoryServiceImpl — 树形结构构建

分类的数据库存储是平的（所有分类在同一张表中，通过 parentId 关联），但前端需要树形结构。用 Java Stream 在内存中构建树：

// 第一步：查出所有分类（最多几十条，全量加载）
List<Category> categories = categoryMapper.selectList(wrapper);

// 第二步：按 parentId 分组
Map<Long, List<CategoryVO>> childrenMap = vos.stream()
    .filter(vo -> vo.getParentId() != null)
    .collect(Collectors.groupingBy(CategoryVO::getParentId));

// 第三步：取出顶级分类，挂载子分类
List<CategoryVO> tree = vos.stream()
    .filter(vo -> vo.getParentId() == null)   // 顶级
    .peek(vo -> vo.setChildren(childrenMap.getOrDefault(vo.getId(), List.of())))
    .toList();

为什么在内存中构建树而不是用数据库递归查询？ 分类数据量小（通常是几十条），全部加载到内存只需一次查询。用 Stream 构建树的复杂度是 O(n)，远快于数据库的多次递归查询。而且这还有一个额外好处——可以把整个树缓存到 Redis。

删除保护 — 两道防线

删除分类前必须检查：① 有没有子分类（有则拒绝）；② 有没有文章关联该分类（有则拒绝）。这是数据完整性的最后防线——即使数据库外键约束未配置或被禁用，代码层面的校验仍然生效。

4.4 CommentServiceImpl — 评论系统

XSS 防护 — Jsoup.clean()

评论是用户输入的内容，可能包含恶意脚本。用 Jsoup.clean(content, Safelist.none()) 在存储前清洗 HTML 标签——任何 HTML 都会被剥离，只保留纯文本。

5.1 FileServiceImpl — 文件上传安全

文件上传是安全风险最高的功能之一。这个 ServiceImpl 的安全策略：

// ★ 双重白名单校验
private static final Set<String> IMAGE_EXTENSIONS = Set.of(
    "jpg", "jpeg", "png", "gif", "webp", "svg"
);

// ① 扩展名校验 — 文件后缀必须在白名单中
String ext = FilenameUtils.getExtension(originalName).toLowerCase();
if (!IMAGE_EXTENSIONS.contains(ext)) throw ...;

// ② MIME 类型二次校验 — 防止把 .jsp 改成 .jpg 绕过扩展名校验
String contentType = file.getContentType();
if (contentType == null || !contentType.startsWith("image/")) throw ...;

为什么扩展名 + MIME 双重校验？ 攻击者可以把一个可执行文件改名为 virus.jpg 上传。单独的扩展名校验会被绕过。MIME 类型由文件内容决定，比扩展名更可信。两层校验叠加，安全性显著提升。

文件存储路径使用 UUID 重命名：{uploadPath}/images/{yyyy/MM}/{uuid}.{ext}。UUID 防冲突 + 防猜测（攻击者无法遍历文件名）+ 按日期分目录（防止单目录文件过多）。

7.2 为什么 RefreshToken 存在 Redis 而 AccessToken 不存在？

AccessToken（不存 Redis）

15 分钟过期，每次 API 请求都要验证。如果每次查 Redis，Redis 成为瓶颈。JWT 自包含 userId+role，本地验签即可。登出后通过黑名单（Token 的 SHA-256 哈希）处理，但黑名单中的 Token 最多存活 15 分钟。

RefreshToken（存 Redis）

7 天过期，需要支持"主动吊销"。用户在"我授权的设备"页面点击"踢下线"，后端直接删除 Redis 中该用户的 RefreshToken——该设备下次刷新时发现 Redis 中不存在，强制重新登录。

八Redis 缓存设计深度解析

8.1 Cache-Aside 模式的完整实现

Cache-Aside（旁路缓存）是最经典的缓存模式。核心思想：应用程序自己管理缓存的读写，缓存层不和数据库层直接交互。

Cache-Aside 读写流程
读操作
① 先查 Redis
↓ 命中 → 直接返回
↓ 未命中
② 查 MySQL
↓
③ 写入 Redis（设 TTL）
↓
④ 返回数据
写操作
① 删除缓存
↓
② 更新 MySQL
↓
③ 模糊删除相关缓存
↓
④ 延迟 500ms 再删一次（双删）

8.2 延迟双删 — ArticleServiceImpl 的实际代码

// updateArticle() 中的延迟双删
// 第一次删除：更新 DB 前
redisTemplate.delete(RedisKeys.CACHE_ARTICLE + id);

// 更新数据库
articleMapper.updateById(article);

// 清理分页缓存（因为文章变了，列表页也要刷新）
invalidatePageCache();

// 第二次删除：500ms 后再删一次
try { Thread.sleep(500); } catch (...) { ... }
redisTemplate.delete(RedisKeys.CACHE_ARTICLE + id);

为什么删两次？第一次删除后到第二次删除之间的 500ms，如果有并发读请求到来，它发现缓存为空，会去查 DB（此时 DB 已经更新完毕），然后把最新数据写入缓存。第二次删除是兜底——如果有一个读请求在第一次删除之前就已经查了旧数据，在第一次删除之后把它写入缓存，第二次删除会清理这个脏数据。500ms 是经验值，足够大部分并发读完成"查 DB → 写缓存"的流程。

8.3 为什么不同数据用不同 TTL？

缓存类型TTL原因
文章详情1 小时文章写完很少改动，1 小时延迟可接受
前台分页列表10 分钟新文章发布后应在 10 分钟内显示在列表
热门文章30 分钟浏览量变化慢，30 分钟更新一次足够
分类树2 小时分类几乎不改，2 小时很保守
标签列表2 小时同上

不同 TTL 天然防止缓存雪崩。如果所有缓存都用同一个 TTL（比如 1 小时），那每到整点所有缓存同时过期，瞬间所有请求打向数据库——这就是缓存雪崩。不同 TTL 让缓存过期时间错开，流量分担到不同时刻。

8.4 为什么要用 Map 包装缓存值？

Redis 序列化使用 GenericJackson2JsonRedisSerializer 并启用了 DefaultTyping.NON_FINAL。但 JSON 反序列化 ArrayList 时可能因为泛型擦除丢失类型信息。用 Map<String, Object> 包装一层（{"data": [...]}），Jackson 可以正确保留内部泛型类型。这是 JSON 序列化框架在处理 Java 泛型时的常见 workaround。

8.5 为什么有搜索条件时跳过缓存？

搜索关键词是用户任意输入的，组合几乎无穷无尽。缓存这些结果命中率极低，反而会占满 Redis 内存。所以 ArticleServiceImpl 中判断 if (StringUtils.hasText(query.getKeyword())) → 不走缓存。这是"缓存不值得"的经典场景。

九MyBatis-Plus 工程化实践

9.1 LambdaQueryWrapper — 类型安全的革命

传统 MyBatis 的字符串列名有两个痛点：① 拼写错误编译期不报错；② 数据库改字段名时，IDE 无法自动重构所有引用。

// 传统写法（字符串列名）
wrapper.eq("categroy_id", categoryId);  // 拼错了！编译不报错，运行时才发现

// Lambda 写法（方法引用）
wrapper.eq(Category::getCategoryId, categoryId);  // 编译报错：找不到 getCategroyId

9.2 分页插件的内幕

MyBatis-Plus 的 PaginationInnerInterceptor 拦截了 MyBatis 的 Executor：

  1. 检测到 SQL 是 SELECT 语句且参数中有 Page 对象
  2. 先执行 COUNT 查询（自动去掉 ORDER BY 和 SELECT 字段，只保留 FROM 和 WHERE）
  3. 把 COUNT 结果写入 Page.total
  4. 在原 SQL 末尾追加 LIMIT offset,size
  5. 执行分页查询，结果写入 Page.records

9.3 MetaObjectHandler — 自动填充的原理

每次 MyBatis-Plus 执行 insert 或 update 时，MetaObjectHandler 拦截并检查 Entity 中标注了 @TableField(fill = FieldFill.INSERT) 的字段，自动调用 fill 方法设置值。这样就不需要在每个 Service 方法里手动 article.setCreateTime(LocalDateTime.now())。

9.4 为什么用 BeanUtil.copyProperties？

ArticleServiceImpl.createArticle() 中用了 Hutool 的 BeanUtil.copyProperties(dto, article)。因为 ArticleDTO 和 Article 有大量同名同类型字段（title、contentMd、coverImage 等），逐个 set 只会产生大量机械代码。但注意：copyProperties 之后再手动设置默认值（status=DRAFT、viewCount=0 等），因为 DTO 中没有这些字段。

十项目工程化思想总结

统一返回格式

R<T> 保证前端对接成本为 O(1)。如果 25 个接口返回 25 种格式，前端要写 25 套解析逻辑。

统一异常处理

Controller 不写 try-catch。异常统一抛 → @RestControllerAdvice 统一拦截 → 统一转 R。异常日志集中管理。

三层数据对象

Entity(DB) ≠ DTO(入参) ≠ VO(出参)。各层变化互不影响，接口向后兼容。

Controller 薄如纸

接参数 → 调 Service → 返回 R。业务逻辑全在 Service，事务边界全在 Service。

配置集中管理

版本号在父 POM，Redis Key 在 RedisKeys，状态码在 ResultCode。改一处即改全局。

模块边界清晰

Maven 多模块 = 编译期强制执行代码边界。不会出现"写着写着就相互引用"的失控。
